# 📜 Скрипти лабораторної роботи: PKI

Чотири bash-скрипти що супроводжують лабораторну роботу.  
Запускаються **по порядку** — кожен будується на результатах попереднього.

---

## ⚡ Швидкий старт

```bash
# Клонувати репозиторій
git clone https://github.com/YOUR_USERNAME/pki-lab.git
cd pki-lab/scripts

# Надати права на виконання
chmod +x *.sh

# Запускати по порядку
bash 01_ssh_keys.sh
bash 02_caesar.sh
bash 03_rsa_demo.sh
bash 04_create_ca.sh
```

## 01_ssh_keys.sh — Мережа та SSH-ключі

### Що робить

1. Показує IP-адресу VM та шлюз (`ip a`, `ip route`)
2. Запитує IP вашої VM і перевіряє доступність (`ping`)
3. Генерує пару ключів **ED25519** у `~/.ssh/id_ed25519_lab`
4. Показує публічний ключ та його відбиток (fingerprint)
5. Додає публічний ключ до `~/.ssh/authorized_keys`
6. Тестує підключення до localhost по ключу (без пароля)

### Запуск

```bash
bash 01_ssh_keys.sh
```

### Очікуваний результат

```
══════════════════════════════════════════
  КРОК 1 — Перевірка мережі
══════════════════════════════════════════
  ℹ Отримані IP-адреси:
     192.168.1.105/24
  ℹ Шлюз за замовчуванням:
     192.168.1.1

  Введіть IP вашої VM: 192.168.1.105
  ✔ VM 192.168.1.105 доступна

...

  ✔ Автентифікація по ключу працює
```

### Файли що створює

```
~/.ssh/
├── id_ed25519_lab      ← приватний ключ
├── id_ed25519_lab.pub  ← публічний ключ
└── authorized_keys     ← сюди додано публічний ключ
```

### Навіщо це в реальному житті

> SSH-ключі замість паролів — стандарт для будь-якого production-сервера.  
> GitHub, GitLab, AWS, Ansible — всі використовують саме цей механізм.

## 02_caesar.sh — Шифр Цезаря

### Що робить

1. Пояснює принцип шифру Цезаря з таблицею алфавіту
2. **Розшифровує завдання від викладача** — `SDWULD HW KRQRU` (зсув 3)
3. Показує покрокове розшифрування кожної літери
4. Дозволяє зашифрувати власний текст з довільним зсувом
5. Ламає зашифровану фразу **брутфорсом** — перебирає всі 25 ключів

### Запуск

```bash
bash 02_caesar.sh
```

### Інтерактивна частина

Скрипт запитає:
```
  Введіть текст для шифрування: SLAVA UKRAINI
  Введіть зсув (1-25): 7
```

### Очікуваний результат (фрагмент)

```
══════════════════════════════════════════
  ЧАСТИНА 4 — Злам брутфорсом (всі 25 варіантів)
══════════════════════════════════════════
  Зсув 1:  RCVTKC GV JQPQT
  Зсув 2:  QBUSJ BFU IPOPO
  ✔ Зсув 3:  PATRIA ET HONOR  ← ось він!
  Зсув 4:   OZSQHZ DS GMNMQ
  ...
```

### Навіщо це в реальному житті

> Демонструє головну ідею криптографії: стійкість забезпечується **розміром ключового простору**, а не секретністю алгоритму. Цезар — 25 ключів. AES-256 — 2²⁵⁶ ключів.

## 03_rsa_demo.sh — RSA: шифрування та підпис

### Що робить

1. Генерує RSA приватний ключ (2048 біт) і витягує публічний
2. **Шифрує файл** публічним ключем → `message.enc`
3. **Розшифровує** приватним ключем → порівнює з оригіналом
4. Створює документ і **підписує** його приватним ключем
5. Перевіряє підпис на оригіналі → `Verified OK`
6. Підробляє документ і перевіряє → `Verification Failure`
7. Демонструє **лавинний ефект** хешів (зміна 1 символу → інший хеш)

### Запуск

```bash
bash 03_rsa_demo.sh
```

### Очікуваний результат (фрагмент)

```
══════════════════════════════════════════
  КРОК 5 — Перевірка підпису
══════════════════════════════════════════
  ℹ Перевірка оригінального документа...
  ✔ ПІДПИС ДІЙСНИЙ — документ не змінено

  ℹ Перевірка підробленого документа...
  ✔ ПІДПИС НЕ ДІЙСНИЙ — підробку виявлено ✓
```

### Файли що створює

```
~/rsa_demo/
├── private.pem      ← RSA приватний ключ
├── public.pem       ← RSA публічний ключ
├── message.txt      ← оригінальне повідомлення
├── message.enc      ← зашифроване повідомлення
├── message_dec.txt  ← розшифроване повідомлення
├── order.txt        ← документ для підпису
├── order.sig        ← цифровий підпис
└── order_fake.txt   ← підроблений документ (підпис не пройде)
```

### Навіщо це в реальному житті

> Цифровий підпис — основа ЕЦП, підпису програм (Windows Authenticode), верифікації Docker-образів, підпису Git-комітів.

## 04_create_ca.sh — Власний CA та сертифікати

### Що робить

1. Створює **кореневий CA** (Root Certificate Authority) — ключ 4096 біт, сертифікат на 10 років
2. Генерує ключ і **CSR сервера** (`tsa233.lab`)
3. **CA підписує сертифікат** сервера з SAN (Subject Alternative Names) — включає `tsa233.lab`, `www.tsa233.lab`, `mail.tsa233.lab`, `192.168.1.10`
4. **Перевіряє ланцюг довіри** — `openssl verify`
5. Демонструє що чужий сертифікат **відхиляється**
6. Видає **клієнтський сертифікат** (для mTLS)
7. Якщо nginx встановлено — **автоматично налаштовує HTTPS**

### Запуск

```bash
bash 04_create_ca.sh
```

> ⚠️ Якщо потрібно налаштувати nginx — запускати з `sudo`:
> ```bash
> sudo bash 04_create_ca.sh
> ```

### Очікуваний результат (фрагмент)

```
══════════════════════════════════════════
  КРОК 4 — Перевірка ланцюга довіри
══════════════════════════════════════════
  ℹ Перевірка: чи підписаний server.crt нашим CA?
  ✔ Ланцюг довіри підтверджено: server.crt ← VIIT Root CA

  ℹ Перевірка: прийме чи чужий сертифікат?
  ✔ Чужий сертифікат відхилено — ланцюг довіри не збігається ✓
```

### Файли що створює

```
~/pki/
├── ca/
│   ├── ca.key        ← приватний ключ CA ⚠️ зберігати в безпеці!
│   ├── ca.crt        ← сертифікат CA (імпортувати в браузер)
│   └── ca.srl        ← серійний лічильник виданих сертифікатів
├── server/
│   ├── server.key    ← приватний ключ сервера
│   ├── server.csr    ← запит на підпис
│   ├── server.crt    ← підписаний сертифікат сервера
│   └── san.cnf       ← конфіг Subject Alternative Names
└── client/
    ├── client.key    ← приватний ключ клієнта
    ├── client.csr    ← запит на підпис
    └── client.crt    ← підписаний клієнтський сертифікат
```

### Прибрати попередження браузера

Щоб браузер довіряв сертифікату `tsa233.lab` — імпортувати `~/pki/ca/ca.crt`:

**Firefox:** `Налаштування` → `Конфіденційність і захист` → `Сертифікати` → `Переглянути сертифікати` → `Імпортувати`

**Chrome/Edge:** `Налаштування` → `Безпека` → `Керування сертифікатами` → вкладка `Довірені кореневі центри` → `Імпорт`

### Навіщо це в реальному житті

> Корпоративні PKI використовують саме таку схему: внутрішній Root CA підписує сертифікати для всіх внутрішніх сервісів. Усі корпоративні ПК мають цей CA в довірених — і браузер не показує попереджень.

## ❗ Типові помилки

| Помилка | Причина | Рішення |
|---------|---------|---------|
| `Permission denied` при підключенні по ключу | Неправильні права на `~/.ssh/` | `chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys` |
| `openssl: command not found` | Не встановлено | `sudo apt install openssl` |
| `connect: Connection refused` у скрипті 01 | SSH-сервер не запущено | `sudo systemctl start ssh` |
| `nginx -t` видає помилку | Порт 443 зайнятий або шлях до сертифіката неправильний | Перевірити `sudo ss -tlnp \| grep 443` |
| Браузер показує попередження на `https://tsa233.lab` | CA не імпортовано як довірений | Імпортувати `~/pki/ca/ca.crt` (інструкція вище) |
