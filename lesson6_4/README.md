## Вступне завдання

```
SDWULD HW KRQRU
```

Що це означає? Спробуйте розшифрувати самостійно.

**Алгоритм розшифрування (шифр Цезаря):**

1. Візьміть кожну літеру шифротексту
2. Зсуньте її на **3 позиції назад** за латинським алфавітом
3. Якщо вийшли за межу (A, B, C) — переходьте з кінця: A→X, B→Y, C→Z

```
Шифротекст:   S  D  W  U  L  D     H  W     K  R  Q  R  U
Зсув -3:      P  A  T  R  I  A     E  T     H  O  N  O  R
Відкритий:    P  A  T  R  I  A  _  E  T  _  H  O  N  O  R
```

> Це **шифр Цезаря** — один із найдавніших шифрів (I ст. до н.е.). Гай Юлій Цезар використовував зсув на 3 для листування з полководцями. Сьогодні він не є безпечним, але наочно демонструє принцип **підстановочного шифрування**.

## 1. Поняття криптографії

### 1.1 Визначення та завдання

**Криптографія** (від грец. *kryptós* — прихований, *gráphō* — пишу) — це наука про методи захисту інформації шляхом її перетворення з метою унеможливлення несанкціонованого читання або зміни.

Головні завдання криптографії формулюють через модель **CIA**:

```
┌─────────────────────────────────────────────────────────┐
│                    Трикутник CIA                        │
│                                                         │
│                  Конфіденційність                       │
│                  (Confidentiality)                      │
│                        ▲                                │
│                       / \                               │
│                      /   \                              │
│                     /     \                             │
│    Цілісність ◄────/       \────► Доступність           │
│    (Integrity)              (Availability)              │
└─────────────────────────────────────────────────────────┘
```

| Властивість | Визначення | Засіб забезпечення |
|-------------|------------|-------------------|
| **Конфіденційність** | Доступ лише авторизованим суб'єктам | Шифрування |
| **Цілісність** | Дані не змінено без дозволу | Хеш-функції, ЕЦП |
| **Автентичність** | Підтвердження особи або джерела | Цифровий підпис, сертифікати |
| **Доступність** | Авторизований доступ у будь-який момент | Резервування (не криптографія) |

### 1.2 Основні поняття

| Термін | Визначення |
|--------|------------|
| **Відкритий текст** (plaintext) | Вихідне повідомлення до шифрування |
| **Шифротекст** (ciphertext) | Результат шифрування |
| **Ключ** (key) | Секретний параметр алгоритму |
| **Шифрування** (encryption) | Перетворення plaintext → ciphertext |
| **Дешифрування** (decryption) | Зворотне перетворення ciphertext → plaintext |
| **Криптоаналіз** | Наука про злам криптосистем |
| **Криптостійкість** | Стійкість алгоритму до злому |

### 1.3 Коротка історія

```
│ ~1900 до н.е.   Шифр скитала (Спарта) — транспозиція стрічки
│ ~50 до н.е.     Шифр Цезаря — зсув алфавіту на 3
│ XV ст.          Шифр Віженера — поліалфавітна заміна
│ 1917            Одноразовий блокнот (Вернам) — абсолютна стійкість
│ 1940-і          Enigma — електромеханічний роторний шифр
│ 1977            DES — перший стандарт симетричного шифрування
│ 1978            RSA — перший практичний алгоритм асиметричного шифрування
│ 2001            AES — сучасний стандарт симетричного шифрування
│ 2000-і          TLS/SSL — захист транспортного рівня (HTTPS)
│ Сьогодні        Постквантова криптографія (NIST PQC 2024)
```

### 1.4 Класифікація криптографічних алгоритмів

```
                     Криптографічні алгоритми
                              │
            ┌─────────────────┼─────────────────┐
            │                 │                 │
     Симетричні         Асиметричні        Хеш-функції
     (1 ключ)           (2 ключі)          (без ключа)
            │                 │                 │
    ┌───────┴───────┐  ┌──────┴──────┐  ┌───────┴───────┐
  Потокові     Блокові  Шифрування  ЕЦП  MD5     SHA-256
  (RC4)    (AES, 3DES) (RSA, ECC) (RSA) [слабкий] [стійкий]
```

## 2. Автентичність та цілісність даних

### 2.1 Хеш-функції

**Хеш-функція** — це математична функція, яка перетворює вхідні дані довільної довжини у рядок фіксованої довжини (**дайджест**, або **хеш**).

**Властивості криптографічної хеш-функції:**

1. **Детермінованість** — однакові вхідні дані завжди дають однаковий хеш
2. **Однонаправленість** — за хешем неможливо відновити вхідні дані
3. **Лавинний ефект** — зміна навіть 1 біта змінює хеш кардинально
4. **Стійкість до колізій** — неможливо знайти два різних входи з однаковим хешем

**Приклад:**

```
Вхід: "Привіт, курсанте!"
SHA-256: a3f9d8c1e4b72069f5a8c3d1e6b90247f8c3a5d1e2b7f4c9a0d8e3b1c6f2a5d9

Вхід: "Привіт, курсанте."  ← змінено лише один символ
SHA-256: 7b2e9f1a4c83d0e5b6c2a8f1d4e7b3c0a5d9f2e1b8c4a7d0e3b6f9c2a5d8e1b4
```

**Поширені хеш-алгоритми:**

| Алгоритм | Розмір хешу | Стан | Застосування |
|----------|-------------|------|--------------|
| **MD5** | 128 біт | Скомпрометований | Лише перевірка завантажень (не для безпеки) |
| **SHA-1** | 160 біт | Вразливий | Застарілий, не рекомендується |
| **SHA-256** | 256 біт | Безпечний | TLS, цифрові підписи, Bitcoin |
| **SHA-512** | 512 біт | Безпечний | Хешування паролів, підписи |
| **bcrypt** | Змінний | Безпечний | Виключно для паролів |

**Практичне застосування хешу:**

```bash
# Перевірка цілісності завантаженого файлу
sha256sum ubuntu-22.04-server.iso
# Порівняти з контрольною сумою на сайті виробника

# Хешування в Linux
echo -n "password" | sha256sum
# 5e884898da28047151d0e56f8dc6292773603d0d6aabbdd62a11ef721d1542d8
```

### 2.2 MAC (Message Authentication Code)

**MAC** — це хеш повідомлення, обчислений із використанням **спільного секретного ключа**. Забезпечує одночасно цілісність і автентифікацію джерела.

```
┌────────────┐            ┌─────────────────────┐
│ Відправник │            │    Отримувач        │
│                         │                     │
│ Повідомлення M          │ Повідомлення M'     │
│ + Секретний ключ K      │ + Секретний ключ K  │
│                         │         │           │
│    HMAC(K, M) = T ─────►│    HMAC(K, M') = T' │
│                         │    T == T' ?        │
│                         │    Так → цілісність │
│                         │    підтверджена     │
└─────────┘               └─────────────────────┘
```

**HMAC** (Hash-based MAC) — найпоширеніша реалізація: `HMAC = H(K XOR opad || H(K XOR ipad || M))`

Використовується у: JWT-токенах, IPsec, TLS, API-автентифікації.

### 2.3 Цифровий підпис (ЕЦП)

**Цифровий підпис** — криптографічний механізм, який забезпечує:
- **Автентичність** — підтвердження особи підписувача
- **Цілісність** — виявлення будь-яких змін у документі
- **Незаперечність** — підписувач не може заперечити факт підписання

**Принцип роботи (асиметрична криптографія):**

```
ПІДПИСАННЯ (відправник):
┌────────────────────────────────────────┐
│  Документ M                            │
│      │                                 │
│      ▼                                 │
│  SHA-256(M) = H          ┌─────────┐   │
│      │                   │Приватний│   │
│      └──────────────────►│  ключ   │   │
│                          └────┬────┘   │
│                               │        │
│                          Підпис S      │
└───────────────────────────────┼────────┘
                                │
              Надсилається: (M, S)
                                │
ПЕРЕВІРКА (отримувач):          ▼
┌────────────────────────────────────────┐
│  SHA-256(M) = H'         ┌─────────┐   │
│                          │Публічний│   │
│  Verify(S, публ.ключ) = H│  ключ   │   │
│                          └─────────┘   │
│  H == H' → підпис дійсний              │
└────────────────────────────────────────┘
```

**Алгоритми цифрового підпису:**

| Алгоритм | Основа | Розмір ключа | Застосування |
|----------|--------|--------------|--------------|
| **RSA** | Факторизація чисел | 2048–4096 біт | TLS, S/MIME, PDF-підписи |
| **DSA** | Дискретний логарифм | 1024–3072 біт | SSH-ключі |
| **ECDSA** | Еліптичні криві | 256–521 біт | TLS 1.3, Bitcoin, JWT |
| **Ed25519** | Twisted Edwards крива | 256 біт | SSH, Git-підписи (сучасний вибір) |

### 2.4 Інфраструктура відкритих ключів (PKI)

**PKI** (Public Key Infrastructure) — це сукупність технологій, процедур і організаційних заходів, яка забезпечує **управління цифровими сертифікатами** та вирішує фундаментальну проблему довіри до публічних ключів у відкритих мережах.

#### Проблема, яку вирішує PKI

Уявіть таку ситуацію: ви хочете захищено з'єднатися з `bank.ua`. Сервер надсилає вам свій публічний ключ. Але звідки ви знаєте, що цей ключ справді належить банку, а не зловмиснику, який перехопив з'єднання?

```
БЕЗ PKI — атака "людина посередині" (MITM):

  Ви ──── запит ────► Зловмисник ──── запит ────► bank.ua
  Ви ◄─ "Ось ключ" ─ Зловмисник         (підміна ключа!)
```

> Без PKI неможливо відрізнити справжній ключ від підробленого. Будь-хто може згенерувати пару ключів і стверджувати, що це ключ банку.

**Рішення PKI:** прив'язати публічний ключ до ідентифікатора (домену, особи, організації) через **цифровий сертифікат**, підписаний **довіреною третьою стороною** — Центром сертифікації (CA).

---

#### Ієрархія довіри (Chain of Trust)

PKI будується на принципі **ланцюжка довіри**: довіра передається від кореневого CA вниз по ієрархії.

```
┌──────────────────────────────────────────────────────────┐
│              Root CA (Кореневий центр сертифікації)      │
│                                                          │
│  • Самопідписаний сертифікат (підписує себе сам)         │
│  • Фізично захищений (апаратний модуль HSM, сейф)        │
│  • Зазвичай НЕ видає сертифікати напряму                 │
│  • Вбудований у браузери та ОС (Mozilla, Microsoft)      │
│  • Приклади: DigiCert Root CA, ISRG Root X1 (Let's Enc.) │
└────────────────────────┬─────────────────────────────────┘
                         │ Root CA підписує сертифікат ↓
                         ▼
┌──────────────────────────────────────────────────────────┐
│            Intermediate CA (Проміжний центр)             │
│                                                          │
│  • Його сертифікат підписаний Root CA                    │
│  • Безпосередньо видає сертифікати кінцевим суб'єктам    │
│  • Якщо скомпрометовано — відкликається, Root CA цілий   │
│  • Приклад: DigiCert TLS RSA SHA256 2020 CA1             │
└────────────────────────┬─────────────────────────────────┘
                         │ Intermediate CA підписує сертифікат ↓
                         ▼
┌──────────────────────────────────────────────────────────┐
│            End-Entity сертифікат (bank.ua)               │
│                                                          │
│  • Виданий конкретному домену / особі / серверу          │
│  • Містить публічний ключ власника та підпис CA          │
│  • Обмежений термін дії (зазвичай 1–2 роки)             │
└──────────────────────────────────────────────────────────┘
```

**Чому два рівні, а не Root CA напряму?**
Root CA тримається в режимі **офлайн** у фізично захищеному сховищі. Якщо Intermediate CA скомпрометовано — його відкликають і видають новий, не торкаючись Root CA. Це ізолює ризики та захищає всю ієрархію довіри.

#### Цифровий сертифікат X.509

**X.509** — міжнародний стандарт (ITU-T) формату цифрових сертифікатів. Сертифікат — це **підписаний CA документ**, що підтверджує відповідність публічного ключа ідентифікатору.

```
┌─────────────────────────────────────────────────────────┐
│                    X.509 Сертифікат                     │
├─────────────────────────────────────────────────────────┤
│ Версія:         v3                                      │
│ Серійний №:     0A:1B:2C:3D:4E:5F  (унікальний у CA)   │
│ Алгоритм підп.: SHA256withRSA                           │
├─────────────────────────────────────────────────────────┤
│ Subject (кому виданий):                                 │
│   CN = bank.ua                                          │
│   O  = ПриватБанк АТ                                    │
│   C  = UA                                               │
├─────────────────────────────────────────────────────────┤
│ Issuer (хто видав):                                     │
│   CN = DigiCert TLS RSA SHA256 2020 CA1                 │
│   O  = DigiCert Inc                                     │
├─────────────────────────────────────────────────────────┤
│ Термін дії:                                             │
│   Від:  2024-01-15 00:00:00 UTC                         │
│   До:   2025-02-15 23:59:59 UTC                         │
├─────────────────────────────────────────────────────────┤
│ Публічний ключ:  RSA, 2048 біт                          │
├─────────────────────────────────────────────────────────┤
│ Розширення (Extensions):                                │
│   Subject Alternative Names (SAN):                     │
│     DNS: bank.ua, DNS: www.bank.ua, DNS: api.bank.ua   │
│   Key Usage: Digital Signature, Key Encipherment       │
│   Extended Key Usage: TLS Web Server Authentication    │
│   CRL Distribution Points: http://crl.digicert.com/…  │
│   OCSP: http://ocsp.digicert.com                       │
├─────────────────────────────────────────────────────────┤
│ Підпис CA:                                              │
│   SHA-256 від усього вище, зашифрований приватним       │
│   ключем Intermediate CA                               │
└─────────────────────────────────────────────────────────┘
```

**Як браузер перевіряє сертифікат (покроково):**

```
1. Отримує сертифікат bank.ua від сервера
2. Знаходить підписувача: Intermediate CA DigiCert
3. Перевіряє підпис Intermediate CA на сертифікаті bank.ua
   (розшифровує підпис публічним ключем Intermediate CA,
    порівнює з SHA-256 від тіла сертифіката)
4. Знаходить підписувача Intermediate CA: DigiCert Root CA
5. Знаходить Root CA у вбудованому сховищі довіри ОС/браузера
6. Перевіряє підпис Root CA на сертифікаті Intermediate CA
7. Додаткові перевірки:
   ✓ Термін дії не минув
   ✓ Домен збігається з CN або SAN
   ✓ Сертифікат не відкликаний (CRL / OCSP)
8. Замок у браузері — з'єднання встановлено
```

---

#### Відкликання сертифікатів

Якщо приватний ключ скомпрометовано — сертифікат потрібно відкликати **до закінчення терміну дії**.

| Механізм | Принцип роботи | Особливість |
|----------|----------------|-------------|
| **CRL** (Certificate Revocation List) | CA публікує список серійних номерів відкликаних сертифікатів | Файл може бути великим, оновлюється з затримкою |
| **OCSP** (Online Certificate Status Protocol) | Браузер надсилає запит CA: "цей сертифікат дійсний?" | Залежить від доступності OCSP-сервера |
| **OCSP Stapling** | Сервер сам отримує підписану відповідь OCSP і надсилає клієнту разом із сертифікатом | Сучасний підхід, знімає навантаження з OCSP |

---

#### Типи сертифікатів за рівнем перевірки

| Тип | Що перевіряється CA | Час видачі | Приклад |
|-----|---------------------|------------|---------|
| **DV** (Domain Validation) | Лише контроль над доменом (відповідь на HTTP-запит або DNS-запис) | Хвилини | Let's Encrypt, більшість сайтів |
| **OV** (Organization Validation) | Домен + існування організації у реєстрі | Дні | Корпоративні сайти |
| **EV** (Extended Validation) | Домен + організація + юридична перевірка особи | Тижні | Банки, держоргани |

> **Let's Encrypt** — безкоштовний публічний CA, що видає DV-сертифікати автоматично через протокол ACME. Завдяки йому HTTPS став стандартом для всього інтернету (~400 млн активних сертифікатів).

---

#### Компоненти PKI

| Компонент | Роль |
|-----------|------|
| **CA** (Certificate Authority) | Видає, підписує та відкликає сертифікати |
| **RA** (Registration Authority) | Перевіряє особу заявника перед видачею |
| **CRL / OCSP** | Механізми відкликання скомпрометованих сертифікатів |
| **Сховище сертифікатів** | База даних виданих/відкликаних сертифікатів |
| **Кінцевий суб'єкт** | Власник сертифіката (сервер, особа, пристрій) |

#### Практична робота з сертифікатами

```bash
# Переглянути сертифікат сайту
openssl s_client -connect bank.ua:443 -showcerts </dev/null 2>/dev/null \
  | openssl x509 -text -noout

# Перевірити термін дії сертифіката
openssl s_client -connect bank.ua:443 </dev/null 2>/dev/null \
  | openssl x509 -noout -dates

# Переглянути локальний .crt файл
openssl x509 -in certificate.crt -text -noout

# Перевірити статус через OCSP
openssl ocsp -issuer intermediate.crt -cert bank.crt \
  -url http://ocsp.digicert.com -resp_text
```

## 3. Конфіденційність даних

### 3.1 Симетричне шифрування

**Симетричне шифрування** — один і той самий ключ використовується для шифрування і дешифрування.

```
          Секретний ключ K
               │         │
               ▼         ▼
Відкритий текст → [Шифрування] → Шифротекст → [Дешифрування] → Відкритий текст
```

**Переваги:**
- Висока швидкість (у 100–1000 разів швидше асиметричного)
- Підходить для великих обсягів даних

**Недолік — проблема розподілу ключів:**
> Як безпечно передати спільний ключ, якщо захищений канал ще не встановлено?

#### AES (Advanced Encryption Standard)

Найпоширеніший сучасний алгоритм. Прийнятий NIST у 2001 р.

| Параметр | Значення |
|----------|----------|
| Розмір блоку | 128 біт |
| Розмір ключа | 128, 192 або 256 біт |
| Кількість раундів | 10 / 12 / 14 |
| Структура | Підстановочно-перестановча мережа |

**Режими роботи AES:**

| Режим | Назва | Особливість | Застосування |
|-------|-------|-------------|--------------|
| **ECB** | Electronic Codebook | Кожен блок шифрується незалежно | Не рекомендується (вразливий) |
| **CBC** | Cipher Block Chaining | Кожен блок XOR з попереднім | Файли, диски (з IV) |
| **CTR** | Counter Mode | Блокований шифр → потоковий | Мережа, потоки даних |
| **GCM** | Galois/Counter Mode | CTR + автентифікація (AEAD) | TLS 1.3, HTTPS (найкращий вибір) |

> **AEAD** (Authenticated Encryption with Associated Data) — одночасно шифрує і автентифікує. Це стандарт для сучасних систем.

### 3.2 Асиметричне шифрування

**Асиметричне шифрування** — використовуються два математично пов'язані ключі:
- **Публічний ключ** — відомий усім, використовується для шифрування
- **Приватний ключ** — відомий лише власнику, використовується для дешифрування

```
          Публічний ключ Б       Приватний ключ Б
                  │                      │
                  ▼                      ▼
  Відкритий → [Шифрування] → Шифротекст → [Дешифрування] → Відкритий
    текст                                                     текст

  (Аліса шифрує)                          (Боб дешифрує)
```

**Математична основа RSA:**

Стійкість RSA ґрунтується на складності **факторизації великих чисел**:
- Помножити два великих простих числа — легко: `p × q = n`
- Розкласти `n` на множники — надзвичайно важко при великих `p`, `q`

**Обмін ключами Діффі-Гелмана (DH):**

Вирішує проблему встановлення спільного секрету по відкритому каналу:

```
Аліса                           Боб
  │  Узгодили відкрито: g, p     │
  │                              │
  │  a = 6 (секрет Аліси)        │  b = 15 (секрет Боба)
  │  A = g^a mod p               │  B = g^b mod p
  │                              │
  │────────── A ────────────────►│
  │◄─────────────────── B ───────│
  │                              │
  │  K = B^a mod p               │  K = A^b mod p
  │  (однакові результати!)      │
```

Використовується в: TLS (ECDHE), SSH, IPsec.

### 3.3 Гібридне шифрування

На практиці **симетричне і асиметричне шифрування поєднують**:

```
┌─────────────────────────────────────────────────────────────┐
│                    Гібридна схема (TLS)                     │
│                                                             │
│  1. Асиметричне шифрування для встановлення сеансового ключа│
│     ┌────────┐                              ┌────────┐      │
│     │Клієнт  │──── Публічний ключ сервера ──►│Сервер  │      │
│     │        │◄─── Сеансовий ключ (зашифр.)──│        │      │
│     └────────┘                              └────────┘      │
│                                                             │
│  2. Симетричне (AES-GCM) для шифрування трафіку             │
│     ┌────────┐  ←────── AES(K, дані) ───────► ┌────────┐   │
│     │Клієнт  │  ────────────────────────────── │Сервер  │   │
│     └────────┘        Швидко та безпечно       └────────┘   │
└─────────────────────────────────────────────────────────────┘
```

Саме так працює **HTTPS (TLS 1.3)**:
- RSA або ECDH — для обміну ключами
- AES-256-GCM — для шифрування даних
- SHA-384 — для контрольних сум

### 3.4 Практичне застосування: TLS/SSL

**TLS** (Transport Layer Security) — протокол, що забезпечує конфіденційність і цілісність для будь-якого протоколу прикладного рівня.

**Спрощене TLS 1.3 рукостискання:**

```
Клієнт                                    Сервер
  │                                          │
  │──── ClientHello (TLS 1.3, cipher list)──►│
  │◄─── ServerHello + Certificate ───────────│
  │◄─── (публічний ключ для ECDHE)           │
  │                                          │
  │  [Обидва обчислюють спільний секрет]     │
  │                                          │
  │──── Finished (MAC) ────────────────────►│
  │◄─── Finished (MAC) ──────────────────────│
  │                                          │
  │════ Зашифрований трафік (AES-GCM) ══════│
```

**Версії TLS та їх статус:**

| Версія | Рік | Статус |
|--------|-----|--------|
| SSL 2.0/3.0 | 1995/1996 | Заборонені (критичні вразливості) |
| TLS 1.0 | 1999 | Застарілий, заборонений PCI DSS |
| TLS 1.1 | 2006 | Застарілий, заборонений RFC 8996 |
| **TLS 1.2** | 2008 | Підтримується (з правильними налаштуваннями) |
| **TLS 1.3** | 2018 | Рекомендований, сучасний стандарт |

### 3.5 Шифрування дисків та файлів

**Шифрування дисків** захищає дані при фізичному доступі до накопичувача:

| Інструмент | ОС | Алгоритм | Призначення |
|------------|-----|----------|-------------|
| **LUKS** | Linux | AES-256-XTS | Шифрування розділів/дисків |
| **VeraCrypt** | Cross-platform | AES, Serpent | Зашифровані контейнери |
| **BitLocker** | Windows | AES-256-XTS | Системні диски Windows |

**Практичний приклад — LUKS:**

```bash
# Створити зашифрований розділ
sudo cryptsetup luksFormat /dev/sdb1

# Відкрити зашифрований розділ
sudo cryptsetup open /dev/sdb1 secret_data

# Використовувати як звичайний диск
sudo mkfs.ext4 /dev/mapper/secret_data
sudo mount /dev/mapper/secret_data /mnt/secret
```

### 3.6 Управління ключами

Ключ — найцінніший елемент криптосистеми. **Безпека системи визначається безпекою ключа**, а не алгоритму.

**Принципи управління ключами:**

1. **Генерація** — використовувати криптографічно стійкий генератор псевдовипадкових чисел (CSPRNG)
2. **Зберігання** — апаратні модулі безпеки (HSM), захищені сховища (HashiCorp Vault, AWS KMS)
3. **Розподіл** — передавати тільки по захищених каналах
4. **Ротація** — регулярна заміна ключів
5. **Знищення** — безповоротне видалення після закінчення терміну дії
